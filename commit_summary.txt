Implement conversation persistence for web widget using Redis

## Summary

Successfully implemented a comprehensive conversation persistence feature for Chatwoot's web widget that allows incognito users to maintain conversation continuity when navigating between pages. The implementation uses Redis for session management and maintains backward compatibility with existing functionality.

## Features Implemented

### 1. Redis Infrastructure
- Added visitor mapping keys to `lib/redis/redis_keys.rb`:
  - `VISITOR_CONVERSATION_MAPPING`: Maps visitor ID to conversation tokens
  - `VISITOR_CONTACT_MAPPING`: Maps visitor ID to contact source IDs  
  - `VISITOR_PAGE_INFO`: Stores page info before conversation creation

### 2. VisitorConversationMapping Model
- New Redis-based model with 30-day TTL for visitor session management
- Methods for setting/getting conversation tokens, contact source IDs, and page info
- Comprehensive error handling and logging throughout
- Automatic cleanup on conversation resolution

### 3. Backend Controller Updates

#### Base Controller (`app/controllers/api/v1/widget/base_controller.rb`)
- Added visitor ID handling from X-Visitor-ID header
- Redis conversation lookup for existing conversations
- Enhanced conversation creation with page info from Redis
- Conversation token generation for persistent mapping

#### Conversations Controller (`app/controllers/api/v1/widget/conversations_controller.rb`)
- Logic to check existing conversations before creating new ones
- 'Live chat widget opened' webhook fires only for new conversations (not page navigation)
- Redis cleanup when conversations are resolved
- Page info storage in Redis before conversation creation

#### Website Token Helper (`app/controllers/concerns/website_token_helper.rb`)
- Enhanced contact lookup with Redis fallback for incognito users
- Visitor mapping storage for contact persistence
- Seamless contact creation for new visitors

### 4. Frontend Implementation

#### Utility Functions (`app/javascript/widget/helpers/utils.js`)
- `generateVisitorId()`: Creates unique visitor ID using timestamp and random string
- `getVisitorId()`: Retrieves or creates visitor ID from sessionStorage
- SessionStorage ensures persistence across page navigation within same session

#### API Integration (`app/javascript/widget/helpers/axios.js`)
- Added X-Visitor-ID header to all API requests via interceptor
- Automatic visitor ID injection for all widget communications

#### Endpoint Updates (`app/javascript/widget/api/endPoints.js`)
- Updated all API endpoints to include visitor_id parameter
- Enhanced page info collection and transmission
- Comprehensive referrer and page tracking

### 5. Additional Enhancements
- Updated PowerShell development scripts to fix emoji encoding issues
- Conversation listener for page info extraction from messages
- Comprehensive logging throughout for debugging and monitoring

## Technical Details

### Persistence Strategy
- Uses visitor ID in sessionStorage (persists across page navigation)
- Redis mapping with 30-day TTL for server-side session management
- Graceful fallback to existing behavior when Redis unavailable

### Webhook Behavior
- 'Live chat widget opened' webhook fires only once per session
- Page navigation within same session does not trigger new webhooks
- Maintains existing webhook functionality for new conversations

### Incognito User Support
- Full support for incognito/private browsing modes
- No dependency on cookies or localStorage
- Session-based persistence using sessionStorage

### Backward Compatibility
- Zero breaking changes to existing conversation/message functionality
- Existing authenticated users continue to work as before
- New features are additive and non-destructive
