Fix conversation ID mismatch between frontend and backend with Redis validation

## Summary

Fixed critical conversation persistence issue where frontend and backend were finding different conversation IDs for the same visitor (frontend: 453, backend: 486). This was caused by stale Redis mappings pointing to old or resolved conversations.

## Key Changes

### Backend Improvements
- **Enhanced conversation lookup logic** with Redis mapping validation before use
- **Added conversation ID to tokens** for direct conversation validation
- **Implemented automatic Redis cleanup** when conversations are resolved
- **Added stale mapping detection** and automatic cleanup
- **Improved conversation token generation** with conversation ID inclusion

### Redis Mapping Validation
- **validate_redis_conversation_mapping()** - Validates Redis mappings before use
- **Checks conversation existence** and status (resolved conversations invalidated)
- **Automatic cleanup** of invalid/stale mappings
- **Conversation resolution callback** to clean up Redis mappings

### Conversation Lookup Consistency
- **Unified lookup logic** ensures frontend and backend find same conversation
- **Fallback validation** when Redis mappings are invalid
- **Automatic mapping updates** to maintain consistency
- **Enhanced logging** for debugging conversation lookup flow

- `app/controllers/api/v1/widget/base_controller.rb` - Enhanced conversation lookup with validation
