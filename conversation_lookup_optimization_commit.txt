Optimize Conversation Lookup: Eliminate Redundant Redis Validation and Token Decoding

## Problem Identified
Server logs showed redundant conversation lookup operations during widget initialization:

1. **Multiple token decoding**: Same conversation token decoded multiple times per request
2. **Duplicate validation**: Redis validation and conversation extraction happening separately
3. **Redundant database queries**: Same conversation looked up multiple times
4. **Excessive logging**: Verbose logs making debugging difficult

## Root Cause Analysis
The conversation lookup flow had architectural redundancy:

```
find_conversation_via_redis()
├── validate_redis_conversation_mapping() 
│   ├── Decode token
│   ├── Find contact_inbox
│   ├── Validate conversation
│   └── Return boolean
└── extract_conversation_from_token()
    ├── Decode token AGAIN
    ├── Find contact_inbox AGAIN  
    ├── Look up conversation AGAIN
    └── Return conversation object
```

This caused the same token to be decoded and validated twice per request, creating unnecessary database queries and confusing logs.

## Solution Implemented

### 1. Consolidated Validation and Extraction
**File**: `app/controllers/api/v1/widget/base_controller.rb`

**BEFORE**: Separate validation and extraction methods
```ruby
if validate_redis_conversation_mapping(visitor_id, conversation_token)
  conversation = extract_conversation_from_token(conversation_token)
end
```

**AFTER**: Single combined method
```ruby
conversation = validate_and_extract_conversation_from_token(visitor_id, conversation_token)
```

**Benefits**:
- ✅ Token decoded only once per request
- ✅ Contact_inbox lookup only once per request  
- ✅ Conversation validation and extraction in single operation
- ✅ Cleaner, more efficient code flow

### 2. Enhanced Conversation Extraction Logic
**Method**: `extract_conversation_from_token_data`

**Improvements**:
- **Specific conversation lookup**: Try exact conversation ID first
- **Status validation**: Check if conversation is resolved before using
- **Fallback mechanism**: Use last open conversation if specific one unavailable
- **Redis mapping update**: Update mapping when using fallback conversation
- **Comprehensive logging**: Clear logging for each step without redundancy

### 3. Removed Redundant Methods
**Eliminated**:
- `validate_redis_conversation_mapping()` - Logic moved to combined method
- `validate_conversation_from_token()` - Logic moved to combined method
- `find_conversation_from_token_data()` - Renamed and enhanced as `extract_conversation_from_token_data()`

### 4. Optimized Logging
**File**: `app/controllers/api/v1/widget/conversations_controller.rb`

**Reduced redundant logs**:
- Removed duplicate header logging
- Focused on essential conversation flow information
- Cleaner request tracking with visitor ID

## Technical Implementation Details

### New Combined Flow
```ruby
def validate_and_extract_conversation_from_token(visitor_id, conversation_token)
  # 1. Decode token once
  token_data = decode_token(conversation_token)
  
  # 2. Find contact_inbox once  
  contact_inbox = find_contact_inbox(token_data[:source_id])
  
  # 3. Extract conversation with validation
  conversation = extract_conversation_from_token_data(contact_inbox, token_data)
  
  # 4. Return conversation or nil
  conversation
end
```

### Enhanced Conversation Extraction
```ruby
def extract_conversation_from_token_data(contact_inbox, token_data)
  # Try specific conversation first
  if token_data[:conversation_id].present?
    specific = find_specific_conversation(token_data[:conversation_id])
    return specific if specific&.not_resolved?
  end
  
  # Fallback to last open conversation
  fallback = find_last_open_conversation(contact_inbox)
  update_redis_mapping(fallback) if fallback
  fallback
end
```
