# Dashboard: Webhook Configuration (UI)

This document describes the user interface within the Chatwoot dashboard for managing account-level webhooks. For a detailed understanding of the webhook system's backend architecture, payload structure, and event types, refer to [Backend Webhooks Documentation](./../../backend_architecture/webhooks.md).

## Overview

The Webhooks settings UI allows administrators to configure endpoints to which Chatwoot will send HTTP POST requests when specific events occur within their account. This enables integration with external systems for notifications, data synchronization, or triggering custom workflows.

## Access and UI Components

-   **Route**: `/app/accounts/{account_id}/settings/webhooks`
-   **Main Component**: `app/javascript/dashboard/routes/dashboard/settings/webhooks/Index.vue`. This page lists existing webhooks and allows for their creation, editing, and deletion.
    -   **Webhook List**: Displays configured webhooks, showing the URL, status (active/inactive), and subscribed events.
        -   Component: `WebhooksTable.vue` or similar (e.g., part of `app/javascript/dashboard/components/widgets/Webhooks/`).
    -   **"Add Webhook" / "Create Webhook" Button**: Opens a modal for creating a new webhook configuration.
        -   Modal Component: `AddWebhookModal.vue` or `WebhookForm.vue` (e.g., `app/javascript/dashboard/components/widgets/Webhooks/WebhookForm.vue`).

-   **Webhook Form (Create/Edit Modal)**:
    -   **URL**: The endpoint URL where Chatwoot will send the webhook payload.
    -   **Subscriptions**: A multi-select dropdown or checkboxes to choose the events the webhook should subscribe to (e.g., `message_created`, `conversation_created`, `contact_updated`).
        -   Available events are defined in `Webhook::ALLOWED_EVENTS` (`app/models/webhook.rb`).
    -   **HMAC Mandatory**: Checkbox to enable HMAC signature verification.
    -   **HMAC Token**: If HMAC is mandatory, an input field for the secret token. This token is generated by Chatwoot or can be user-defined.
    -   **Status**: Toggle to activate or deactivate the webhook (though this might be implicit or a separate action).

## Key Functionalities

### 1. Listing Webhooks
-   **Fetching Webhooks**:
    -   Vuex action: `webhooks/get` (`app/javascript/dashboard/store/modules/webhooks.js`).
    -   API: `GET /api/v1/accounts/{account_id}/webhooks` via `WebhooksAPI.get()` (`app/javascript/dashboard/api/webhooks.js`).
    -   Controller: `app/controllers/api/v1/accounts/webhooks_controller.rb#index`.

### 2. Creating a New Webhook
-   **Frontend Process**:
    -   Admin fills out the webhook form (URL, subscriptions, HMAC settings).
    -   Vuex action: `webhooks/create`.
    -   API: `POST /api/v1/accounts/{account_id}/webhooks`.
-   **Backend Process**:
    -   Controller: `app/controllers/api/v1/accounts/webhooks_controller.rb#create`.
    -   Payload: `url`, `subscriptions` (array of event names), `hmac_mandatory` (boolean), `hmac_token` (string).
    -   Creates a new `Webhook` record associated with the account.
    -   Model: `Webhook` (`app/models/webhook.rb`).

### 3. Editing an Existing Webhook
-   **Frontend Process**:
    -   Admin modifies URL, subscriptions, or HMAC settings.
    -   Vuex action: `webhooks/update`.
    -   API: `PATCH /api/v1/accounts/{account_id}/webhooks/{webhook_id}`.
-   **Backend Process**:
    -   Controller: `app/controllers/api/v1/accounts/webhooks_controller.rb#update`.
    -   Updates the specified `Webhook` record.

### 4. Deleting a Webhook
-   **Frontend Process**:
    -   Admin confirms deletion.
    -   Vuex action: `webhooks/delete`.
    -   API: `DELETE /api/v1/accounts/{account_id}/webhooks/{webhook_id}`.
-   **Backend Process**:
    -   Controller: `app/controllers/api/v1/accounts/webhooks_controller.rb#destroy`.
    -   Deletes the `Webhook` record.

### 5. HMAC Security
-   If `hmac_mandatory` is enabled for a webhook:
    -   A secret `hmac_token` is stored (and should be treated as sensitive).
    -   Chatwoot will generate an HMAC SHA256 signature of the JSON payload using this token.
    -   The signature is sent in the `X-Chatwoot-Hmac-Sha256` HTTP header.
    -   The receiving endpoint should verify this signature to ensure the payload authenticity and integrity.
    -   The HMAC token is generated on webhook creation if not provided by the user, or can be regenerated.

## State Management (Vuex)

-   **`app/javascript/dashboard/store/modules/webhooks.js`**:
    -   Manages webhook configurations for the account.
    -   State: `records` (array of webhook objects), `uiFlags`.
    -   Actions: `get`, `create`, `update`, `delete`.
    -   Mutations: `SET_WEBHOOKS_UI_FLAG`, `SET_WEBHOOKS`, `ADD_WEBHOOK`, `EDIT_WEBHOOK`, `DELETE_WEBHOOK`.

## Backend API Endpoints & Models (UI Specifics)

-   **Model**: `Webhook` (`app/models/webhook.rb`)
    -   Attributes directly managed by UI: `url`, `subscriptions` (array of strings), `hmac_mandatory` (boolean), `hmac_token` (string), `status` (enum: `active`, `inactive`).
    -   `account_id`.
-   **Controller**: `app/controllers/api/v1/accounts/webhooks_controller.rb`
    -   Provides CRUD operations for `Webhook` records scoped to the current account.

This UI provides a user-friendly way to manage webhook configurations. The actual dispatch and payload generation are handled by the backend services detailed in the [Backend Webhooks Documentation](./../../backend_architecture/webhooks.md).
