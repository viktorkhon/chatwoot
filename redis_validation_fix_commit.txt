### Problem Solved
Fixed critical Redis validation logic flaw that was causing valid conversation mappings to be incorrectly marked as "stale" and cleared, leading to new conversations being created during page navigation instead of maintaining persistence.

### Root Cause
The `validate_redis_conversation_mapping` method was looking up contact_inbox by source_id from the Redis token, which could return a different contact_inbox than the current `@contact_inbox`, causing validation failures for valid conversations.

### Key Changes

#### 1. Fixed Redis Validation Logic
**File:** `app/controllers/api/v1/widget/base_controller.rb`
- Modified `validate_redis_conversation_mapping` to use current `@contact_inbox` instead of lookup by source_id
- Added source_id consistency check between token and current session
- Enhanced validation with detailed logging of each step

#### 2. Enhanced Debugging Infrastructure
- Added comprehensive logging to track complete conversation lookup flow
- Enhanced `find_conversation_via_redis` with detailed token validation logging
- Improved `find_conversation_via_database` to show all available conversations
- Added logging to `store_conversation_in_redis` to track when conversations are stored

#### 3. Validation Process Improvements
- Enhanced `validate_conversation_from_token` with detailed logging
- Added visibility into conversation existence and status checks
- Improved error handling and validation result tracking
